# PR 合并策略与平台配置

## 为什么要把策略和平台配置放在一起看

很多团队会讨论“我们应该用 squash 还是 rebase merge”，但真正落地时，平台配置往往和口头约定并不一致。

所以这页关注的是两件事：

- 你们想要什么样的主线历史
- 平台到底允许什么合并方式

## 建议先统一 3 个问题

1. 主线想保留 PR 粒度还是提交粒度
2. 已共享历史是否允许更强的重写
3. 平台默认开启哪些合并方式

## 一个更稳的决策框架

- 如果想主线最短、每个 PR 对应一个提交：优先 squash merge
- 如果想保留原提交粒度，同时维持线性：考虑 rebase merge
- 如果想明确保留合并节点和分叉结构：保留 merge commit

真正危险的不是选错一次，而是团队没有统一默认值，导致每个 PR 都按个人偏好落地。

## 平台配置应该对应什么

建议把平台配置和团队规范对齐：

- 只打开团队认可的 merge methods
- 明确分支保护规则
- 明确谁能 bypass，谁不能
- 如果平台支持自动删除分支、合并队列，也一并统一

## 常见误区

- 以为平台默认值就是团队最佳实践
- 口头说用 squash，但平台实际三种都开着
- 只讨论命令，不讨论 PR 平台上的真实入口
