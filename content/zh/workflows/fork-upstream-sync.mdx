# Fork 与上游同步教程

## 这个工作流解决什么问题

在 fork 协作模式里，你通常同时面对两个远端：

- `origin`：你自己的 fork
- `upstream`：官方主仓库

很多同步混乱，不是 Git 命令本身太难，而是这两个远端角色没有分清楚。

这一页的目标，就是把下面这条边界固定下来：

- 从 `upstream` 拉最新主线
- 把同步结果更新到本地
- 再把自己的分支推回 `origin`

## 什么时候适合这个流程

这个流程通常适用于：

- 开源项目贡献
- 公司内部镜像协作
- 没有直接写权限，只能通过 fork + PR 提交改动的仓库

如果你本来就直接推送团队主仓库，那就更适合看普通的功能分支协作流，而不是 fork 工作流。

## 最小稳定配置

第一次配置时，推荐先确认：

```bash
git remote -v
git remote add upstream <upstream-url>
git remote -v
```

理想状态通常是：

- `origin` 指向你自己的 fork
- `upstream` 指向团队或开源项目主仓库

## 推荐顺序

### 1. 先更新上游信息

```bash
git fetch upstream
```

这一步的价值是：先观察官方主线到底变成了什么样，而不是直接改你当前分支。

### 2. 回到本地主分支并同步上游

```bash
git switch main
git rebase upstream/main
```

如果团队不希望重写你本地主分支历史，也可以改用：

```bash
git merge upstream/main
```

### 3. 把同步后的本地主分支推回自己的 fork

```bash
git push origin main
```

这一步很关键，因为 fork 模式下，你后续通常还是从自己的 `origin/main` 再切功能分支。

### 4. 再从更新后的主分支切出工作分支

```bash
git switch -c feature/update-docs
```

## 一个典型的日常节奏

如果你每周都在一个开源项目里提交改动，一个很稳的节奏通常是：

```bash
git fetch upstream
git switch main
git rebase upstream/main
git push origin main
git switch -c feature/small-fix
```

这样做的核心价值是：你的功能分支总是从最新主线切出来，而不是从落后的 fork 主分支上继续叠。

## 关键检查点

每次同步前后，建议至少看这几项：

```bash
git remote -v
git branch -vv
git log --oneline --decorate --graph --all -10
```

重点确认：

1. `origin` 和 `upstream` 没有接反
2. 当前分支是不是你以为的那个分支
3. 本地主分支同步完成后，是否真的已经和 `upstream/main` 对齐

## 常见误区

### 误区 1：把 `origin` 当成上游主仓库

在 fork 工作流里，`origin` 往往是你的个人 fork，不是团队主仓库。

### 误区 2：直接在落后的 fork 主分支上切新分支

这样会把旧历史也一起带进新任务。

### 误区 3：同步 upstream 之后忘了把结果推回 origin

这样你自己的 fork 还停在旧状态，后面再开分支时容易继续从旧基底出发。

## 特殊情况

- 如果默认分支不是 `main`，要把命令中的目标分支改成真实分支名
- 如果你本地主分支已经有自己的额外提交，rebase 或 merge 前先确认这些提交是否应该保留
- 如果团队使用受保护分支，`origin/main` 的更新策略可能也有额外要求

## 相关命令

- `git remote`
- `git fetch`
- `git switch`
- `git rebase`
- `git merge`
- `git push`
