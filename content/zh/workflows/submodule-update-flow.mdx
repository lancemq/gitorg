# 子模块更新流程教程

## 这个流程为什么容易出错

子模块工作流最容易让人困惑的一点是：

你看起来像是在更新一个目录，实际上是在同时处理两层仓库：

- 子模块仓库自身的提交历史
- 主仓库里记录的“子模块指针”

如果不把这两层分开看，就会出现“明明改了子模块，主仓库为什么还显示变化”这种典型困惑。

## 推荐顺序

### 1. 先在子模块里切到目标提交

```bash
git submodule update --init --recursive
cd path/to/submodule
git fetch origin
git switch main
git pull --ff-only
```

这一步的目标，是先确认子模块仓库自己已经到了你想要的状态。

### 2. 再回到主仓库提交子模块指针变化

```bash
cd ../..
git status
git add path/to/submodule
git commit -m "chore: update submodule pointer"
```

主仓库提交的并不是子模块源码本身，而是“指向了哪个子模块提交”。

### 3. 最后统一验证初始化和递归更新

发布给其他开发者之前，最好确认他们按照标准命令拉下来后能否得到一致状态：

```bash
git submodule update --init --recursive
```

## 一个完整的更新示例

```bash
git submodule update --init --recursive
cd path/to/submodule
git fetch origin
git switch main
git pull --ff-only
cd ../..
git status
git add path/to/submodule
git commit -m "chore: update submodule pointer"
```

这套流程最大的价值是：先更新子模块，再明确提交主仓库里的指针变更，中间不混淆。

## 关键检查点

每次更新子模块时，至少确认：

1. 子模块自己是否已经是目标提交
2. 主仓库里是不是只出现了预期的子模块指针变化
3. 初始化和递归更新命令是否仍然可用

实操上建议看：

```bash
git status
git diff --submodule
```

## 常见误区

### 误区 1：以为主仓库提交的是子模块文件内容

通常不是。主仓库提交的是子模块引用到哪个提交。

### 误区 2：只更新了子模块仓库，没有提交主仓库指针

这样别人拉主仓库时，仍然拿不到你想要的子模块版本。

### 误区 3：忘记初始化子模块就开始工作

这会让本地状态看起来“不完整”或者目录为空。

## 特殊情况

- 如果团队锁定子模块只能指向特定版本，需要在更新前先确认版本策略
- 如果子模块本身也在走功能分支，更新主仓库前要先确认子模块分支是否已经稳定
- 如果仓库里有多层子模块，最好始终带上 `--recursive`
