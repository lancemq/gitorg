# Git 快速上手学习路径

## 这篇内容解决什么问题

很多人第一次接触 Git 时，会把命令当成一串要背的咒语。更稳妥的学习方式是先建立一条最小闭环：

1. 拿到仓库
2. 修改文件
3. 暂存改动
4. 生成提交
5. 与远端同步

只要这条闭环能跑通，再去理解 rebase、reflog、引用和对象模型就会容易很多。

## 学完这篇你应该做到什么

如果这篇内容吸收得比较顺，你至少应该能做到下面几件事：

- 知道 `clone`、`add`、`commit`、`fetch`、`pull`、`push` 分别解决什么问题
- 能区分工作区、暂存区、提交历史这三个层次
- 能在本地改完一个文件后，把它安全地提交并同步到远端
- 遇到 `pull`、`push` 报错时，先判断问题是“同步时机”还是“分支状态”

对初学者来说，这四件事比“会不会 rebase”更重要，因为它们构成了后面所有 Git 操作的地基。

## 先建立一个最小心智模型

在真正开始敲命令前，先把 Git 想成三层：

1. 工作区：你当前正在编辑的文件
2. 暂存区：你准备放进下一次提交的内容
3. 提交历史：已经保存下来的快照和提交关系

很多初学者一开始觉得 Git 复杂，往往不是命令太多，而是把这三层混在了一起。

## 开始前的环境准备

建议在第一天就把下面几件事做完：

### 1. 配置身份

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
```

如果不先配置用户名和邮箱，第一次提交时 Git 往往会直接报错。

### 2. 找一个练习仓库

练习阶段最好不要直接在高风险主仓库里乱试。更稳妥的做法是：

- 用一个自己的测试仓库
- 或者 fork / clone 一个专门拿来练习的小项目
- 或者新建一个本地目录，用 `git init` 做实验仓库

### 3. 养成固定检查动作

在每个关键动作前后，都尽量看一眼：

```bash
git status
git log --oneline --decorate -5
```

前者帮助你看清工作区和暂存区，后者帮助你确认历史有没有按预期变化。

## 建议的第一阶段命令

### 1. `git clone`

用途：把远端仓库复制到本地，并自动记录默认远端 `origin`。

```bash
git clone <repository-url>
```

学习重点：

- 本地仓库不只是文件副本，也包含完整历史
- `origin` 只是默认远端名字，不是特殊关键字
- clone 完成后，先看 `git status` 和 `git branch -vv`

常见场景：

- 新加入团队，第一次把仓库拉到本地
- 新电脑或新工作目录，需要重新建立本地副本
- 想在不影响原目录的情况下重新做一套干净实验

### 2. `git add`

用途：把当前工作区改动加入暂存区。

```bash
git add .
```

学习重点：

- 工作区、暂存区、提交历史是三个不同层次
- `git add` 不是“提交”，而是“准备提交内容”
- 可以只暂存部分文件，而不是一次把所有内容都提交

常见场景：

- 你改了多个文件，但这次只想提交其中一部分
- 你先保存代码，再决定下一次提交要表达哪个意图
- 你希望提交粒度更小，方便 review 和回滚

### 3. `git commit`

用途：把暂存区内容保存成一个新的提交。

```bash
git commit -m "feat: add login form"
```

学习重点：

- 提交应该表达一个明确意图
- 提交信息要让未来的自己和同事都能看懂
- 频繁、小粒度提交通常比一次大提交更安全

常见场景：

- 完成一个清晰的小功能或一次局部修复
- 想把当前阶段成果先保存下来，避免继续修改时丢失上下文
- 准备在提交后再做下一轮对比、同步或代码审查

### 4. `git fetch`

用途：只下载远端最新引用和对象，不改本地当前分支。

```bash
git fetch origin
```

学习重点：

- fetch 更像“先观察，再决定”
- 它适合在合并或 rebase 之前先同步远端状态
- 如果你经常担心 `pull` 改坏当前分支，先养成 fetch 的习惯

常见场景：

- 你想先看看同事刚推了什么
- 你本地还有改动，不想直接让远端变化进来
- 你准备在同步前先判断到底该 merge 还是 rebase

### 5. `git pull`

用途：先 fetch，再把远端变更整合进当前分支。

```bash
git pull --ff-only
```

学习重点：

- pull 不是一个原子概念，它本质上是“下载 + 整合”
- 整合阶段可能是 fast-forward、merge、rebase 或 squash
- 初学阶段优先使用 `--ff-only`，更不容易在不知情时生成 merge commit

常见场景：

- 你当前分支明确要和上游保持同步
- 你已经确认工作区干净，并且能接受这次整合动作
- 你想在开始今天工作前先把最新状态带下来

### 6. `git push`

用途：把本地提交发布到远端。

```bash
git push origin main
```

学习重点：

- push 失败通常意味着远端比你更新
- 推送前先确认你要发布的是不是正确分支
- 对共享分支的强推要格外谨慎

常见场景：

- 你已经完成本地提交，准备把结果发布出去
- 你想触发远端 CI、代码评审或备份流程
- 你要把本地特性分支推到远端，供他人查看或协作

## 一条最值得反复练的基本流程

如果你今天只练一次，我更建议练下面这条：

```bash
git clone <repository-url>
cd <repository-name>
git status

# 修改一个文件
git add .
git status
git commit -m "docs: update quick start notes"
git fetch origin
git pull --ff-only
git push origin main
```

这条流程故意不复杂，但它能把下面几件事串起来：

- 工作区变更如何进入暂存区
- 暂存区如何变成提交
- 同步远端前为什么值得先看状态
- 本地提交如何最终发布到远端

## 按天拆分的练习节奏

### 第 1 天：先跑通单人闭环

```bash
git clone <url>
git status
git add .
git commit -m "docs: update README"
git push
```

练习目标：

- 先把“文件改了 -> 成功提交 -> 成功发布”走通一次
- 不追求快，重点是每一步都看清 `git status`

### 第 2 天：开始区分 fetch 和 pull

```bash
git fetch origin
git log --oneline --graph --decorate --all
git pull --ff-only
```

练习目标：

- 看懂 fetch 只更新远端跟踪引用
- 体验 pull 会继续推进当前分支状态
- 观察 `origin/main` 和本地 `main` 的关系变化

### 第 3 天：补上分支与同步

- 创建特性分支
- 提交改动
- fetch 观察远端变化
- 决定 merge 还是 rebase

练习目标：

- 从“一个人写代码”升级到“为团队协作做准备”
- 开始理解为什么不同同步方式会影响历史表达

## 新手最容易卡住的四个问题

### 1. 为什么 `git add` 之后文件还在目录里？

因为 `add` 不是移动文件，而是把当前文件内容记进暂存区，工作区里的文件当然还在。

### 2. 为什么 `git commit` 之后远端还没变化？

因为 commit 只是在本地生成提交，远端是否变化要看你有没有执行 `git push`。

### 3. 为什么 `git push` 会失败？

最常见的原因是：

- 远端比你新
- 你推错了分支
- 当前分支还没有正确建立上游关系

遇到这种情况，不要直接强推，先看：

```bash
git status
git branch -vv
git fetch origin
```

### 4. 为什么 `git pull` 会让我紧张？

因为它不是“只下载”，而是“下载后继续整合”。如果你不想立刻整合，应该优先 `fetch`。

## 初学者最常见的误区

### 误区 1：把 `git add` 当作“提交”

不是。`add` 只是在选择下一次提交要包含什么内容。

### 误区 2：默认把 `git pull` 当安全操作

不是。pull 会真正改变当前分支，尤其是在分支已经分叉时。

### 误区 3：提交前不看 `git status`

这会导致无关文件被误提交。建议把 `git status` 变成每次提交前的固定动作。

### 误区 4：push 失败时直接强推

这会把“我没同步好远端”升级成“我可能覆盖了别人历史”。快速上手阶段先远离强推，优先学会判断分支状态。

## 快速上手阶段的边界

这篇内容刻意没有把所有 Git 命令都塞进来，是因为快速上手阶段最重要的是先稳定，而不是一次学全。

现阶段你可以先不急着碰：

- `rebase`
- `reset --hard`
- `cherry-pick`
- `reflog` 的深度恢复
- 各类 plumbing 命令

不是这些命令不重要，而是它们更适合建立在“最小闭环已经稳定”的基础上。

## 一个更稳的学习判断原则

当你不确定下一步该做什么时，先问自己这三个问题：

1. 我现在看到的是工作区问题、暂存区问题，还是历史问题？
2. 我是想先观察远端，还是想立刻整合远端？
3. 这次操作只影响我自己，还是会影响团队共享分支？

只要你愿意先回答这三个问题，很多 Git 操作都会自然变得清晰。
