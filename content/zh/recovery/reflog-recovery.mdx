# reflog 恢复手册

## 先记住一个结论

很多 Git 误操作之后，最先丢的是“名字”，不是“对象”。所以排查恢复时，第一反应通常不是慌着重做，而是先看 reflog。

## 常见适用场景

- `reset --hard` 之后后悔了
- `rebase` 过程中历史被改乱了
- `pull` 之后想回到原来位置
- 分支被删了，但怀疑提交对象还在

## 恢复步骤

### 1. 先查看 reflog

```bash
git reflog
```

找到误操作前的那条位置记录，比如 `HEAD@{1}` 或 `HEAD@{2}`。

### 2. 不要直接覆盖，先建救援分支

```bash
git checkout -b rescue HEAD@{1}
```

这样最稳，因为你先把旧位置接住，再决定后面怎么整理。

### 3. 再决定真正恢复策略

- 想把当前分支回到旧位置：`git reset --hard <target>`
- 想保留两边内容：从救援分支继续整理
- 想只拿回部分改动：`cherry-pick` 或手动摘取

## 为什么先建分支更安全

因为恢复动作本身也可能继续移动引用。先建一个 `rescue` 分支，等于先把旧位置固定下来，后面怎么试都更安心。

## 一个实用原则

看到历史不对时，优先做这三件事：

1. `git reflog`
2. 建救援分支
3. 再做 reset、merge、cherry-pick 等整理动作
