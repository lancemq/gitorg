# git branch 教程

## 一句话理解

`git branch` 负责管理分支引用，本质上是在查看或移动那些“指向提交的名字”。

## 查看分支

```bash
git branch
```

它会列出本地分支，当前分支前面会有 `*` 标记。

### 查看远端跟踪分支

```bash
git branch -r
```

### 同时查看本地和远端

```bash
git branch -a
```

### 查看上游和最后一次提交

```bash
git branch -vv
```

这是日常排查“我这个分支追踪谁、落后了多少”的高频命令。

## 创建分支

```bash
git branch feature/login
```

这只会创建分支，不会自动切换过去。想创建后直接进入，通常会搭配 `git switch -c`。

## 重命名和删除

```bash
git branch -m old-name new-name
git branch -d feature/login
```

其中 `-d` 只删除已经安全合并的分支；如果你明确要强制删除，才使用 `-D`。

## 它和远端跟踪分支的关系

很多人会把 `main` 和 `origin/main` 混在一起。其实前者是本地分支，后者是远端跟踪分支。fetch 更新的是后者，merge 或 rebase 才会把这些变化整合进前者。

## 常见误区

### 误区 1：branch 会帮我切过去

不会。`git branch` 主要管理分支本身，切换通常交给 `git switch` 或旧式的 `checkout`。

### 误区 2：删除分支等于删除提交

不一定。只要还有别的引用指向这些提交，或者 reflog 还保留着它们，提交对象通常依然存在。

## 一个团队建议

把 `git branch -vv` 养成日常习惯，会大幅减少“我以为自己同步了主分支”这类误判。


## 这条命令在流程里解决什么问题

`git branch` 更接近工作区、暂存区和当前分支这三层状态。理解这类命令时，最重要的不是背参数，而是先判断它会不会改动文件内容、索引快照或当前引用。

## 典型用例

- 把 `git branch` 放进“修改 -> 检查 -> 暂存 -> 提交”的日常循环中，控制当前状态。
- 在切换上下文、整理提交或恢复文件时，用 `git branch` 精准处理工作区与索引。
- 在团队协作前，先通过 `git branch` 把本地状态整理干净，减少后续冲突成本。

## 特殊情况与边界

- 这类命令往往直接碰工作区或暂存区，所以真正的风险不在“命令长什么样”，而在你是否分清它会改哪一层。
- 如果当前目录里还有未备份改动，执行 `git branch` 前先决定这些改动是要提交、stash 还是丢弃。
- 和 `git status` 搭配读，会更容易判断 `git branch` 执行前后到底改变了什么。
- 删除分支前先确认该分支是否已经合并，以及是否仍有远端或 worktree 在使用它。
