# git tag 教程

## 一句话理解

`git tag` 用来给某个提交打上稳定名字，最常见的场景是版本发布。

## 常见类型

- 轻量标签
- 附注标签

附注标签更适合正式发布，因为它包含更多元数据。

## 基本用法

```bash
git tag v1.0.0
git tag -a v1.0.0 -m "Release v1.0.0"
```

## 推送标签

```bash
git push origin v1.0.0
git push origin --tags
```


## 这条命令在流程里解决什么问题

`git tag` 更接近工作区、暂存区和当前分支这三层状态。理解这类命令时，最重要的不是背参数，而是先判断它会不会改动文件内容、索引快照或当前引用。

## 典型用例

- 把 `git tag` 放进“修改 -> 检查 -> 暂存 -> 提交”的日常循环中，控制当前状态。
- 在切换上下文、整理提交或恢复文件时，用 `git tag` 精准处理工作区与索引。
- 在团队协作前，先通过 `git tag` 把本地状态整理干净，减少后续冲突成本。

## 特殊情况与边界

- 这类命令往往直接碰工作区或暂存区，所以真正的风险不在“命令长什么样”，而在你是否分清它会改哪一层。
- 如果当前目录里还有未备份改动，执行 `git tag` 前先决定这些改动是要提交、stash 还是丢弃。
- 和 `git status` 搭配读，会更容易判断 `git tag` 执行前后到底改变了什么。
- 轻量标签和附注标签的语义不同，发布流程里通常更推荐附注标签。
