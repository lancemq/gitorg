# git switch 教程

## 一句话理解

`git switch` 是专门负责“切分支”的命令，它把这项职责从旧式的 `git checkout` 中拆了出来。

## 为什么需要 switch

`checkout` 既能切分支，也能恢复文件，语义过于宽。官方后来提供 `switch` 和 `restore`，就是为了让“切换分支”和“恢复路径”这两类操作更不容易混淆。

## 最常见用法

### 切到已有分支

```bash
git switch feature/login
```

### 创建并切换到新分支

```bash
git switch -c feature/login
```

这和旧写法 `git checkout -b feature/login` 的目的相同，但更容易理解。

## 切回上一个分支

```bash
git switch -
```

在两个分支之间快速来回切换时非常高频。

## 当 Git 拒绝切换时

如果当前工作区改动会在切换时被覆盖，Git 会直接阻止操作。这是一种保护机制，不是故障。

常见处理方式有三种：

1. 先提交
2. 先 stash
3. 你明确知道可以丢弃时，使用 `--discard-changes`

## detached HEAD 也可能通过 switch 进入

```bash
git switch --detach <commit>
```

这适合临时检查历史提交。只要你在这个状态下产生了值得保留的新提交，记得马上新建分支把它接住。

## switch 和 checkout 的边界

- `switch`：切分支
- `restore`：恢复文件
- `checkout`：旧式多用途命令

对于新内容体系或团队培训材料，优先教 `switch` 往往更清晰。


## 这条命令在流程里解决什么问题

`git switch` 更接近工作区、暂存区和当前分支这三层状态。理解这类命令时，最重要的不是背参数，而是先判断它会不会改动文件内容、索引快照或当前引用。

## 典型用例

- 把 `git switch` 放进“修改 -> 检查 -> 暂存 -> 提交”的日常循环中，控制当前状态。
- 在切换上下文、整理提交或恢复文件时，用 `git switch` 精准处理工作区与索引。
- 在团队协作前，先通过 `git switch` 把本地状态整理干净，减少后续冲突成本。

## 特殊情况与边界

- 这类命令往往直接碰工作区或暂存区，所以真正的风险不在“命令长什么样”，而在你是否分清它会改哪一层。
- 如果当前目录里还有未备份改动，执行 `git switch` 前先决定这些改动是要提交、stash 还是丢弃。
- 和 `git status` 搭配读，会更容易判断 `git switch` 执行前后到底改变了什么。
- 切换分支前如果 Git 提示会覆盖本地改动，不要硬闯，先决定是提交、stash 还是丢弃。
