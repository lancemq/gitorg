# git rm

`git rm` 不只是删文件，它还会把“删除”这件事放进暂存区，准备进入下一次提交。

## 常见写法

```bash
git rm app/old-file.ts
git rm -r legacy-folder
git rm --cached .env.local
```

## 三种最常见场景

### 删除文件并提交删除

```bash
git rm notes.txt
git commit -m "remove old notes"
```

### 只从 Git 跟踪中移除，但保留本地文件

```bash
git rm --cached .env.local
```

这在把文件加入 `.gitignore` 时很常见。

### 递归删除目录

```bash
git rm -r tmp/
```

## 为什么它和直接 `rm` 不一样

你可以先用系统 `rm` 删除文件，再用 `git add -u` 或 `git commit -a` 让 Git 感知删除；但 `git rm` 会把“删除文件”和“暂存删除记录”一起完成。

## 风险点

- `git rm` 删除的是已跟踪文件
- 如果还没提交，通常还能从历史或工作区恢复
- 一旦提交并推送，删除就会进入共享历史


## 这条命令在流程里解决什么问题

`git rm` 更接近工作区、暂存区和当前分支这三层状态。理解这类命令时，最重要的不是背参数，而是先判断它会不会改动文件内容、索引快照或当前引用。

## 典型用例

- 把 `git rm` 放进“修改 -> 检查 -> 暂存 -> 提交”的日常循环中，控制当前状态。
- 在切换上下文、整理提交或恢复文件时，用 `git rm` 精准处理工作区与索引。
- 在团队协作前，先通过 `git rm` 把本地状态整理干净，减少后续冲突成本。

## 特殊情况与边界

- 这类命令往往直接碰工作区或暂存区，所以真正的风险不在“命令长什么样”，而在你是否分清它会改哪一层。
- 如果当前目录里还有未备份改动，执行 `git rm` 前先决定这些改动是要提交、stash 还是丢弃。
- 和 `git status` 搭配读，会更容易判断 `git rm` 执行前后到底改变了什么。
