# git add 教程

## 一句话理解

`git add` 会把你选中的改动放进暂存区，供下一次提交使用。

## 为什么暂存区很关键

Git 不是“改完文件就直接提交”，而是先把要提交的内容放进 index，然后再生成 commit。这也是为什么提交可以做到更干净、更有边界。

## 基本用法

```bash
git add README.md
git add src/
git add .
```

## 精细暂存

```bash
git add --patch
```

这能让你按 hunk 选择改动，是拆小提交非常重要的能力。

## 一个实用判断

如果你发现 `git add .` 总把不相干的改动一起带进去，就说明你应该更多使用按文件或按 patch 的暂存方式。


## 这条命令在流程里解决什么问题

`git add` 更接近工作区、暂存区和当前分支这三层状态。理解这类命令时，最重要的不是背参数，而是先判断它会不会改动文件内容、索引快照或当前引用。

## 典型用例

- 把 `git add` 放进“修改 -> 检查 -> 暂存 -> 提交”的日常循环中，控制当前状态。
- 在切换上下文、整理提交或恢复文件时，用 `git add` 精准处理工作区与索引。
- 在团队协作前，先通过 `git add` 把本地状态整理干净，减少后续冲突成本。

## 特殊情况与边界

- 这类命令往往直接碰工作区或暂存区，所以真正的风险不在“命令长什么样”，而在你是否分清它会改哪一层。
- 如果当前目录里还有未备份改动，执行 `git add` 前先决定这些改动是要提交、stash 还是丢弃。
- 和 `git status` 搭配读，会更容易判断 `git add` 执行前后到底改变了什么。
