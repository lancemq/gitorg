# git log 教程

## 一句话理解

`git log` 是查看提交历史的核心命令，既能看简单列表，也能看图结构、作者、时间和差异摘要。

## 日常最常用写法

```bash
git log --oneline --graph --decorate --all
```

这能更直观地展示分支和 merge 关系。

## 常见筛选

```bash
git log -5
git log --author="Alice"
git log -- path/to/file
```

## 为什么 log 很重要

它不只是“看过去发生了什么”，也是理解分支状态、回滚目标和 review 边界的重要工具。


## 这条命令在流程里解决什么问题

从流程位置看，`git log` 更像“先观察、再决定”的命令。它通常不直接改写历史，而是帮助你确认工作区、暂存区、引用或提交对象当前到底处于什么状态。

## 典型用例

- 在真正修改历史或文件之前，先用 `git log` 收集当前状态、差异或证据。
- 把 `git log` 放进代码评审、排障和复盘流程，帮助团队对齐上下文。
- 当你需要解释“现在仓库为什么会这样”时，让 `git log` 先输出可验证的信息。

## 特殊情况与边界

- 大多数观察命令本身不改历史，但它看到的结果会受 HEAD、路径、提交范围或引用选择影响。
- 如果 `git log` 的输出和预期不一致，先确认你看的到底是工作区、索引、当前分支，还是某个历史提交。
- 把 `git log` 和 `git status`、`git log`、`git show` 组合使用，通常比单独看一个输出更稳。
