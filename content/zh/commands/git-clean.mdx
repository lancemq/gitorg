# git clean 教程

## 一句话理解

`git clean` 用来删除未跟踪文件和目录，它不会作用于已经被 Git 跟踪的文件内容。

## 为什么它是高风险命令

因为被删除的未跟踪文件通常不在 Git 历史里，删掉后不容易恢复。

## 最重要的习惯

先 dry-run：

```bash
git clean -n
```

确认无误后再真正执行：

```bash
git clean -f
git clean -fd
```

## 常见误区

### 误区 1：clean 和 reset --hard 一样

不一样。`reset --hard` 主要处理已跟踪文件的工作区状态，`clean` 则处理未跟踪文件和目录。


## 这条命令在流程里解决什么问题

`git clean` 更多是在管理工作副本、目录布局或临时状态，而不只是处理单个提交。它适合解决“我怎样并行工作、怎样暂存现场、怎样控制本地磁盘上的可见状态”这类问题。

## 典型用例

- 在多任务并行、临时切换上下文或清理本地环境时，用 `git clean` 管理工作副本。
- 把 `git clean` 放进团队的日常协作规范里，减少“本地现场太脏、切任务太慢、目录太重”的问题。
- 在大型仓库、子模块或多工作树场景里，用 `git clean` 降低切换成本和认知负担。

## 特殊情况与边界

- 这类命令经常直接改变本地文件系统布局或可见文件范围，所以执行前要先判断是否会影响当前开发中的文件。
- 如果仓库里还有未处理改动，先确认这些改动是要跟着一起迁移、临时保存，还是直接清理掉。
- 当命令同时影响多个目录或嵌套仓库时，最好把操作范围先缩到一个最小实验场景。
- 实际删除前先跑 `git clean -n` 或 `git clean -nd`，确认会被清掉的文件范围。
