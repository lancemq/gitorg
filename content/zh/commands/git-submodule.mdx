# git-submodule 教程

## 一句话理解

git-submodule 用于管理嵌入仓库中的子模块依赖。

## 什么时候适合用

- 当你需要管理嵌入仓库中的子模块依赖
- 当你想把这类操作做成稳定流程而不是手工重复
- 当你需要更准确地理解 Git 在这一步到底记录了什么

## 基本示例

```bash
git submodule update --init --recursive
```

## 读这条命令时最该注意什么

这类命令通常会改变仓库布局或协作方式，团队内最好先统一约定。

## 一个更稳的实践建议

先用小范围实验仓库验证流程，再推广到主仓库或自动化。

## 补充理解角度

- 分发代码或历史
- 降低克隆和检出范围
- 在多仓库或多工作树场景下协作


## 这条命令在流程里解决什么问题

`git submodule` 更多是在管理工作副本、目录布局或临时状态，而不只是处理单个提交。它适合解决“我怎样并行工作、怎样暂存现场、怎样控制本地磁盘上的可见状态”这类问题。

## 典型用例

- 在多任务并行、临时切换上下文或清理本地环境时，用 `git submodule` 管理工作副本。
- 把 `git submodule` 放进团队的日常协作规范里，减少“本地现场太脏、切任务太慢、目录太重”的问题。
- 在大型仓库、子模块或多工作树场景里，用 `git submodule` 降低切换成本和认知负担。

## 特殊情况与边界

- 这类命令经常直接改变本地文件系统布局或可见文件范围，所以执行前要先判断是否会影响当前开发中的文件。
- 如果仓库里还有未处理改动，先确认这些改动是要跟着一起迁移、临时保存，还是直接清理掉。
- 当命令同时影响多个目录或嵌套仓库时，最好把操作范围先缩到一个最小实验场景。
- 子模块状态不是“自动跟着父仓库走”，更新后还要提交父仓库里记录的子模块指针。
