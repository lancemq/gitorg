# Fork and Upstream Sync

## Where this workflow fits

In a fork workflow you usually manage two remotes:

- `origin`: your fork
- `upstream`: the canonical repository

Most confusion comes from blurring those two roles.

## Suggested sequence

### 1. make sure the remotes are mapped correctly

```bash
git remote -v
git remote add upstream <upstream-url>
git remote -v
```

### 2. fetch upstream first

```bash
git fetch upstream
```

### 3. sync your local main branch

```bash
git switch main
git rebase upstream/main
```

If your team prefers not to rewrite local history, use merge instead.

### 4. push the updated main back to your fork

```bash
git push origin main
```

## Why this order is safer

It fixes the remote boundary:

- read from upstream
- stabilize locally
- publish back to origin

## Common mistakes

- treating `origin` as the canonical repo
- cutting new branches from an outdated fork main
- forgetting to push the synced result back to your fork
