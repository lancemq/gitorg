# git log Tutorial

## The short version

`git log` is the main command for reading commit history.

## High-value daily form

```bash
git log --oneline --graph --decorate --all
```

This makes branch structure and merge points much easier to read.


## What problem this command solves in a workflow

In the workflow, `git log` is mostly a “inspect first, decide second” command. It usually does not rewrite history by itself; instead, it helps you confirm the current state of the working tree, index, refs, or commit objects.

## Typical use cases

- Use `git log` before changing files or history so you have observable evidence first.
- Put `git log` into review, debugging, and incident-analysis flow so the team can align on the same context.
- When you need to explain why the repository looks the way it does, let `git log` surface verifiable information first.

## Special cases and boundaries

- Most inspection commands do not mutate history, but their output still depends on which HEAD, path, range, or ref you asked them to inspect.
- If `git log` shows something unexpected, first verify whether you are looking at the working tree, the index, the current branch, or a historical commit.
- Combining `git log` with `git status`, `git log`, and `git show` is usually safer than trusting a single output in isolation.
