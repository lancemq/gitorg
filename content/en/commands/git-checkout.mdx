# git checkout Tutorial

## The short version

`git checkout` is the classic multi-purpose Git command that can both switch branches and restore paths.

## Why it feels overloaded

Because it does two different jobs, newer Git versions introduced `git switch` for branch movement and `git restore` for path recovery.

## Two common uses

### Switch branches

```bash
git checkout main
git checkout -b feature/login
```

### Restore a path

```bash
git checkout -- README.md
```

That restores the file in the working tree from the index, similar to what `git restore README.md` expresses more clearly today.

## Checking out a commit

```bash
git checkout <commit>
```

This puts you into detached HEAD, which is useful for inspection but easy to misuse if you forget to create a branch for work you want to keep.

## Why you still need to understand it

- many older tutorials still teach it
- old scripts and internal docs still use it
- it helps explain where `switch` and `restore` came from


## What problem this command solves in a workflow

`git checkout` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git checkout` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git checkout` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git checkout` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git checkout`.
- Running `git status` before and after `git checkout` is often the fastest way to confirm what really changed.
- `git checkout` mixes branch switching and path restoration, so always inspect whether the last argument is a ref or a path.
