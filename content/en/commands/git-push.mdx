# git push Tutorial

## The short version

`git push` publishes local refs and objects to a remote repository.

## Common forms

```bash
git push origin main
git push -u origin feature/login
git push --force-with-lease
```

`--force-with-lease` is the safer force option for collaborative work.


## What problem this command solves in a workflow

`git push` is about how your local repository exchanges information with remotes, patches, or transportable artifacts. The mental model becomes clearer once you separate download, publish, integrate, and distribute.

## Typical use cases

- Use `git push` when synchronizing upstream state, publishing local work, or packaging history for review or transfer.
- Place `git push` inside collaboration flow so the team can control when information is fetched, integrated, or exposed.
- Use `git push` in automation, mirroring, offline transfer, or email-based workflows when Git history needs to move between systems.

## Special cases and boundaries

- A common remote-workflow mistake is treating download, integration, and publication as the same action. Split them first, then choose flags.
- If the result may affect shared branches or externally visible history, double-check the remote, the target ref, and the permission boundary before executing `git push`.
- Looking at `git fetch`, `git push`, and `git remote -v` together usually makes it easier to tell whether the problem is transport, auth, or ref selection.
- If you really need to rewrite published history, prefer `--force-with-lease` over a blind `--force`.
