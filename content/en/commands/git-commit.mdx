# git commit Tutorial

## The short version

`git commit` turns staged content into a new commit object at the tip of the current branch.

## Common usage

```bash
git commit -m "feat: add search dialog"
git commit --amend
```

Use `--amend` mainly for local cleanup before the history is shared.


## What problem this command solves in a workflow

`git commit` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git commit` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git commit` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git commit` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git commit`.
- Running `git status` before and after `git commit` is often the fastest way to confirm what really changed.
