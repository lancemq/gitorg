# git tag Tutorial

## The short version

`git tag` gives a stable name to a specific commit, most often for releases.

## Common forms

```bash
git tag v1.0.0
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin --tags
```


## What problem this command solves in a workflow

`git tag` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git tag` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git tag` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git tag` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git tag`.
- Running `git status` before and after `git tag` is often the fastest way to confirm what really changed.
- Lightweight and annotated tags serve different purposes, and release workflows usually benefit more from annotated tags.
