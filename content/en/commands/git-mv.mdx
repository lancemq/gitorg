# git mv

`git mv` renames or moves a tracked file and stages that path change at the same time.

## Common examples

```bash
git mv old-name.ts new-name.ts
git mv docs/guide.md docs/intro.md
```

## When it helps most

- during rename-heavy refactors
- when you want file movement and staging to happen in one step

## How to think about it

Git still reasons primarily about content plus paths. `git mv` is mainly a convenience command that keeps your working tree and index aligned cleanly.

## Useful reminder

Git can often infer renames even if you use your shell `mv` first. The main value of `git mv` is clarity and staging convenience.


## What problem this command solves in a workflow

`git mv` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git mv` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git mv` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git mv` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git mv`.
- Running `git status` before and after `git mv` is often the fastest way to confirm what really changed.
