# git switch Tutorial

## The short version

`git switch` is the dedicated command for moving between branches or creating a new branch and entering it immediately.

## Why switch exists

`git checkout` used to cover both branch switching and file restoration. Git later introduced `switch` and `restore` to make those two jobs easier to understand.

## Common usage

### Switch to an existing branch

```bash
git switch feature/login
```

### Create and switch in one step

```bash
git switch -c feature/login
```

This is the clearer modern form of `git checkout -b feature/login`.

## Jump back to the previous branch

```bash
git switch -
```

This is very handy when you alternate between two branches during review or debugging.

## When Git refuses to switch

If local changes would be overwritten, Git aborts the operation as a safety feature. Your usual options are to commit, stash, or explicitly discard changes if that is really what you intend.

## Detached HEAD is also possible

```bash
git switch --detach <commit>
```

That is valid for temporary inspection or experiments. If the resulting commits matter, create a branch right away so they are easy to keep.


## What problem this command solves in a workflow

`git switch` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git switch` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git switch` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git switch` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git switch`.
- Running `git status` before and after `git switch` is often the fastest way to confirm what really changed.
- If Git warns that switching would overwrite local changes, decide whether to commit, stash, or discard before continuing.
