# git restore Tutorial

## The short version

`git restore` is for restoring file content to a known state. It focuses on paths, not on moving branch pointers.

## Why restore exists

Older Git workflows used `git checkout -- <path>` for many file-recovery tasks. Git later introduced `restore` so that “switch branches” and “restore file content” would be expressed by different commands.

## Two main jobs

### Restore the working tree

```bash
git restore README.md
```

This discards unstaged working tree changes and restores the file from the index.

### Unstage a file

```bash
git restore --staged README.md
```

This resets the staged version back to `HEAD`, which makes it a clear way to undo `git add`.

## Restore both index and working tree

```bash
git restore --staged --worktree README.md
```

Use this when you want to unstage and discard the local working copy together.

## Restore from a specific commit

```bash
git restore --source=HEAD~1 src/app.ts
```

That lets you recover one path from another commit without moving the entire branch.

## restore vs reset

- `restore` focuses on file state
- `reset` can affect the index and may also move HEAD

If your real question is “How do I get this file back?”, start with `restore`.


## What problem this command solves in a workflow

`git restore` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git restore` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git restore` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git restore` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git restore`.
- Running `git status` before and after `git restore` is often the fastest way to confirm what really changed.
- When `git restore` touches both staged and working tree content, pay close attention to `--staged`, `--worktree`, and `--source`.
