# git clone Tutorial

## The short version

`git clone` copies an existing repository locally, including its history, objects, refs, and default remote configuration.

## Basic usage

```bash
git clone https://example.com/repo.git
git clone https://example.com/repo.git my-project
```

## What clone sets up

- a full local repository
- a default remote usually named `origin`
- an initial checkout of the default branch


## What problem this command solves in a workflow

`git clone` is about how your local repository exchanges information with remotes, patches, or transportable artifacts. The mental model becomes clearer once you separate download, publish, integrate, and distribute.

## Typical use cases

- Use `git clone` when synchronizing upstream state, publishing local work, or packaging history for review or transfer.
- Place `git clone` inside collaboration flow so the team can control when information is fetched, integrated, or exposed.
- Use `git clone` in automation, mirroring, offline transfer, or email-based workflows when Git history needs to move between systems.

## Special cases and boundaries

- A common remote-workflow mistake is treating download, integration, and publication as the same action. Split them first, then choose flags.
- If the result may affect shared branches or externally visible history, double-check the remote, the target ref, and the permission boundary before executing `git clone`.
- Looking at `git fetch`, `git push`, and `git remote -v` together usually makes it easier to tell whether the problem is transport, auth, or ref selection.
