# git add Tutorial

## The short version

`git add` moves selected changes into the staging area so they can become part of the next commit.

## Common forms

```bash
git add README.md
git add src/
git add .
git add --patch
```

`--patch` is especially important when you want clean, focused commits.


## What problem this command solves in a workflow

`git add` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git add` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git add` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git add` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git add`.
- Running `git status` before and after `git add` is often the fastest way to confirm what really changed.
