# git rm

`git rm` does more than delete a file. It removes the tracked path and stages that deletion for the next commit.

## Common patterns

```bash
git rm app/old-file.ts
git rm -r legacy-folder
git rm --cached .env.local
```

## Typical scenarios

- delete a tracked file and commit that deletion
- stop tracking a file while keeping it locally with `--cached`
- remove whole tracked directories with `-r`

## Why it differs from plain `rm`

You can delete a file with your shell and later stage the deletion another way. `git rm` simply bundles the file removal and staging step together.

## Practical caution

Once committed and shared, the deletion becomes part of project history, so use it deliberately for tracked paths.


## What problem this command solves in a workflow

`git rm` sits close to the working tree, the index, and the current branch. The most important question is not “which flag exists,” but “which layer will this command modify?”

## Typical use cases

- Use `git rm` inside the daily loop of edit, inspect, stage, and commit so local state stays intentional.
- Reach for `git rm` when you need to prepare a clean handoff between file changes, staging decisions, and branch movement.
- Use `git rm` to clean up local state before collaboration steps that are more expensive to recover from later.

## Special cases and boundaries

- The risk in state-changing commands is usually not the syntax itself, but whether you correctly identified which layer they will touch.
- If there are still valuable local edits in the working tree, decide whether they should be committed, stashed, or discarded before running `git rm`.
- Running `git status` before and after `git rm` is often the fastest way to confirm what really changed.
