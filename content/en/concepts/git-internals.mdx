# Git Internals Guide

## The short version

Git is not just a list of file versions. At its core, it is a content-addressable object database, with references layered on top to turn those objects into readable history and collaborative workflows.

## Why internals matter

Many Git commands feel confusing only because the internal model is missing. Once the model is clearer, several behaviors stop feeling mysterious:

- why a commit creates new objects
- why creating a branch is almost instant
- why rebase produces new commit IDs
- why reset can move you back without immediately erasing old objects
- why reflog can help after mistakes

## 1. Git is a content-addressable object database

Pro Git explains that Git stores data in a key-value style object store. Content is written into `.git/objects`, and Git gives it back an object ID derived from that content.

That has two big consequences:

1. Git fundamentally tracks content, not just filenames
2. unchanged content leads to stable object identity

## 2. The four common object types

### blob

A blob stores file content, not the filename.

### tree

A tree records directory structure and points to blobs and other trees.

### commit

A commit normally points to:

- a tree representing the snapshot
- one or more parent commits
- author, timestamp, and message metadata

### tag

A tag gives a human-readable name to an object, often for releases.

## 3. What really happens during commit

Internally, a commit is much more than writing a log line:

1. Git writes a tree from the staged content
2. Git creates a new commit object
3. that commit points to the tree and parent commit(s)
4. the current branch ref moves to the new commit
5. HEAD follows that updated branch ref

So commit is really “add a new node and move a ref forward.”

## 4. A branch is just a movable reference

This is one of the most important Git ideas. A branch is not a full copy of the repository. It is simply a named pointer to a commit.

That explains why:

- creating branches is cheap
- switching branches is fast
- deleting a branch does not instantly delete the underlying commits

## 5. HEAD tells Git where you currently are

In the normal case, `HEAD` is a symbolic ref that points to the current branch, which then points to a commit.

In detached HEAD, `HEAD` points directly to a commit instead. That is why commits made there can become awkward to find unless you create a branch for them.

## 6. History is a graph, not just a list

Git history is better understood as a graph of commits.

### normal commits

Most commits have one parent, so they appear linear.

### merge commits

Merge commits often have two parents, recording the join point of two lines of development.

### after rebase

Rebase does not edit old commits in place. It recreates equivalent changes on a new base, so the graph is expressed through new commit objects.

## 7. Why rebase rewrites history

Because a commit object includes parent information, changing the parent means changing the commit identity. Even if the content looks similar, rebased commits are new objects.

That is why rebase is useful for local cleanup but risky on already-shared history.

## 8. What reset changes

From an internal perspective, reset mainly moves refs, and depending on the mode, it may also resync the index and working tree.

- `--soft`: move HEAD and the current branch
- `--mixed`: also reset the index
- `--hard`: also overwrite the working tree

The old objects usually do not disappear immediately. They often just lose an easy-to-find name.

## 9. Fetch, push, and remote-tracking refs

It helps to separate these names:

- `main`: your local branch
- `origin/main`: your local record of the remote branch

`fetch` updates the second one, not the first. That separation is what gives Git its useful pause between “learn the remote state” and “mutate my branch.”

## 10. Why packfiles matter

The Pro Git packfiles chapter explains how Git stores and transfers objects efficiently in packed form. The practical lesson is that Git is not naïvely copying every full repository state on every operation. It relies on object reuse, compression, and packing to manage history efficiently.

## 11. A useful three-layer mental model

You can think of Git in three layers:

### content layer

blob, tree, commit, and tag objects

### pointer layer

branches, tags, HEAD, and remote-tracking refs

### working layer

the working tree, index, and the command experience you interact with daily

A lot of Git confusion comes from mixing these layers together.

## 12. Re-reading common commands through this model

### `git add`

Moves file state toward the next tree and commit by updating the index.

### `git commit`

Creates new objects and advances the current branch.

### `git merge`

Adds a new history node that joins two lines of development.

### `git rebase`

Recreates commits on top of a different parent chain.

### `git checkout` / `git switch`

Changes what HEAD follows and refreshes the visible working state.

### `git restore`

Restores path content from known object state without necessarily moving a branch.

## 13. Why internals help in practice

Internals are not just trivia. They make judgment better:

- a merge commit becomes an obvious graph join point
- detached HEAD becomes a ref-state issue, not repository corruption
- reset and rebase become easier to reason about in terms of refs and recovery

## The most useful conclusion

Many Git commands are really doing some combination of three things:

1. creating or reusing objects
2. moving references
3. syncing the visible working and staged state

Once those are mentally separate, Git becomes much easier to reason about.
